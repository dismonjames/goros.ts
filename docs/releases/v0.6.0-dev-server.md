# v0.6.0 - Dev Server Speed & HMR-lite

## Highlights

Boronix v0.6.0 introduces fast development reload ("HMR-lite") to improve the developer loop.

- Stable file watching for pages, layouts, route modules, shared server code, and public assets.
- Automatic browser refresh through a lightweight Server-Sent Events channel.
- Route add/remove/rename detection with isolated child restarts.
- Template cache invalidation without process restart.
- Server module reload via isolated child-process restart.
- Safe config and environment file reload handling.
- Watch event debouncing and ignore rules to prevent reload loops.
- Reload latency logs and watch diagnostics (`--debug-watch`).
- Bun and Node development runtime coverage.

## HMR-lite Definition

Boronix uses three reload layers:

1. **Template reload** — `page.html`, `layout.html`, `error.html`, `not-found.html` changes invalidate template cache and trigger browser refresh. No process restart.
2. **Server module reload** — `page.ts`, `api.ts`, `actions.ts`, `middleware.ts`, `app/server/**`, `app/shared/**`, `app/db/**` changes restart an isolated child. Browser reconnects and refreshes.
3. **Structural reload** — route add/remove/rename, `boronix.config.ts`, `.env` changes restart that child.

Boronix uses full-page refresh. Browser state is not preserved.

## Supported File Changes

| File | Behavior |
|------|----------|
| `page.html` | Template cache invalidate, browser refresh |
| `layout.html` | Template cache invalidate, browser refresh |
| `error.html` | Template cache invalidate, browser refresh |
| `not-found.html` | Template cache invalidate, browser refresh |
| `page.ts` | App generation reload, browser refresh |
| `api.ts` | App generation reload, browser refresh |
| `actions.ts` | App generation reload, browser refresh |
| `middleware.ts` | App generation reload, browser refresh |
| `app/server/**` | App generation reload, browser refresh |
| `app/shared/**` | App generation reload, browser refresh |
| `app/db/**` | App generation reload, browser refresh |
| `public/**` | Browser refresh (served from disk) |
| Route directory add/remove | Route tree rescan, browser refresh |
| `boronix.config.ts` | Config reload + route rescan, browser refresh |
| `.env` / `.env.local` | Config reload + route rescan, browser refresh |

## Browser Refresh Behavior

Boritonix injects a small `<script>` dev client into HTML responses during `boronix dev`. The client connects to an SSE endpoint at `/__boronix/dev-events`. When a reload event is broadcast, the browser does a full `window.location.reload()`.

- One batch of file changes produces at most one browser reload.
- The dev client is not injected into JSON, static assets, redirects, or non-HTML responses.
- The dev client is not present in production (`boronix start`).
- The SSE endpoint does not exist in production.

## New CLI Flags

```bash
boronix dev --no-reload       # Disable browser auto-refresh and file watcher
boronix dev --debug-watch     # Print detailed file watch diagnostics
```

## Bun and Node Support

Both runtimes support the full dev reload flow:

- File watcher
- Route reload
- Browser SSE
- Template reload
- Module reload

## Known Limitations

- Browser state is not preserved across reloads.
- Form input that has not been submitted will be lost on full-page refresh.
- Inline script state is reset on refresh.
- Config/env changes may trigger a controlled app reload rather than a hot swap.
- Shared server module changes reload the entire app generation.
- Network filesystem watcher behavior depends on OS/runtime.
- No CSS-only hot injection in this phase.
